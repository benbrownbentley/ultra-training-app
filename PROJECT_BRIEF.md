## The problem

Existing training plans don't work for Ben: they get stuck in Excel sheets, are hard to follow
day-to-day, and don't adapt as the runner's situation changes during training. Ben has built many
plans that died this way.

## The vision

**Vert** is a web app that generates an adaptive endurance training plan — daily gym + running — and
modifies itself as the user logs progress, using an LLM to generate, regenerate, and refine.

## Users

- **v1:** Just Ben (single user, no auth)
- **v2+:** Public sign-ups, each user gets their own private plans

## Mobile strategy (decided 2026-05-17, refined 2026-05-18)

Vert is **mobile-first in intent** — users will log workouts on their phone during or immediately after activity. However, v1 and v2 are built in **Next.js as a mobile-first responsive web app (PWA-ready)**, not a native iOS/Android app. A native mobile app (Expo / React Native) is deferred to v3 or later.

### Strategic reframing (locked in 2026-05-18)

The instinct to "go native immediately like Strava" was reconsidered. Strava's native app is irreplaceable because its killer feature is **live activity tracking** — GPS, heart rate, background sensors, watch integration. Vert's killer feature is different: the **adaptive AI training plan**. That lives on the server and is accessible from any client.

The strategic implication: **Vert may not need to build a tracker at all.** Most serious endurance athletes already track with Garmin, Apple Watch, Wahoo, or Coros and sync to Strava. Vert can do the same — be the brain that produces plans and ingests completed activities from devices people already use. This is why the v1 `source` field (`manual` / `strava` / `garmin` / `apple_health`) was added to workout-log records — it's the seam that lets device sync land later without a destructive migration.

### Why not native now

- **Scope explosion for a learning project.** Native means learning React Native (different from Next.js: different navigation, styling, deployment), Apple's App Store review process, Google Play, push notification infrastructure, and dual iOS/Android build pipelines — on top of learning autonomous coding agents and the web stack.
- **Desktop dev workflow is faster for the learning phase.** Next.js iterates in a browser; native requires simulators, device builds, and longer feedback loops.
- **The current stack is portable.** Supabase has a full React Native SDK, the Claude API is pure HTTP, and TypeScript/React knowledge transfers directly. Logic written cleanly now moves to native later with zero backend changes.

### Architectural discipline to apply now

These decisions cost nothing today and preserve the option to add native cleanly later:

1. **Mobile-first responsive design.** Every screen designed and tested at phone width first. Tailwind + layout discipline — no separate stack needed. (Already reflected in the Today / Plan / Journal / Profile tab specs above.)
2. **PWA-ready Next.js.** Users can install to home screen, get an app-style icon, get most of the "feels like an app" benefit. Configure as a PWA in v2.
3. **Clean logic/UI separation.** Business logic in `lib/` stays framework-agnostic — Supabase queries, Claude API calls, plan-generation helpers. Page components consume `lib/` but don't contain logic. This is the single most valuable architectural rule for future portability.
4. **API-first thinking.** Treat the Next.js server actions as an API the future React Native app will also call. Don't bury logic in page-only contexts.
5. **Device sync over native tracking.** Garmin Connect, Apple HealthKit, and Strava all have APIs/import paths. v2+ work prioritises ingesting completed workouts from those, not building a native GPS recorder.

### When to revisit native

Triggers that would justify the v3 Expo migration:
- Users specifically ask to **track workouts inside Vert** rather than syncing from elsewhere
- **Reliable push notifications** become essential (iOS web push is limited)
- **App Store presence** becomes a credibility / marketing / discoverability play
- Wearable integrations need direct device access (HealthKit on iOS, etc.)

### Migration path when v3 arrives

1. Add an Expo app alongside the Next.js app in the repo (monorepo with Turborepo)
2. Copy `lib/` wholesale — all business logic, Supabase queries, Claude API calls move over unchanged (this is why the lib/ discipline now matters)
3. Rebuild the UI layer using React Native components (View, Text, Pressable instead of div, p, button)
4. Swap Tailwind for NativeWind (same utility-class syntax, React Native compatible)
5. Replace Next.js App Router with Expo Router (conceptually similar)

**Estimated UI migration effort:** 3–6 weeks for an app of this size, assuming clean logic/UI separation.

### Future native features unlocked by migration

- Apple Health / Google Fit integration (direct, not via web import)
- Background GPS / activity tracking (only if users ask for in-app tracking)
- Push notifications (native, not web push)
- Offline-first with background sync
- Wearable integrations (Apple Watch, Garmin)

## Product decisions (locked in during kickoff session, 2026-05-15)

- **Entry point:** An intake wizard collects race goal + athlete profile + constraints before
  generating a plan. This replaces the earlier "hardcoded plan" step.
- **Activity scope:** Endurance sports only. Ultra trail running in v1 (Ben is the target
  user — training for ultras himself). Future versions expand to the full endurance
  multi-sport set: running, swimming, cycling, triathlon, and Ironman, using the same
  data model. Reference target for future scope: Ascent Training's product surface.
- **Workout explanations:** Static glossary for workout types + Claude-generated per-workout
  "why" explaining the purpose of each specific session (Option C of three considered).
- **Physio notes:** Structured input (diagnosis, restrictions, prescribed exercises) in v1.
  Physio as a separate user role with its own login is a v3 feature.
- **Drag-and-drop rescheduling:** Stretch goal for v1. Deferred to v2 if it adds complexity.
- **LLM strategy:** Anthropic Claude is the default for v1 and v2. Fine-tuning / SLM /
  open-source alternatives are parked in the future backlog.
- **Analytics:** PostHog wired in for v2.

### Tab contents (strawman agreed 2026-05-17 — not yet designed)

- **PLAN** — full training block view (locked in 2026-05-17 — see "Plan tab anatomy"
  section below for the full spec). Phase visualization, weekly progression sparkline,

  Direct manual edits (drag-and-drop) live here when shipped (v2) — drag is a manual

  cleanly separated concepts. **Month/calendar view (WEEK / MONTH toggle): v2 backlog.**
- **JOURNAL** — unified surface for everything the user tells Claude. Locked in
  2026-05-17 (see "Journal tab anatomy" section below for the full spec). Four
  entry types in v1: **Note** (freeform, bottom sheet), **Injury** (structured
  form, full route), **Physio** (structured form, full route), **Travel**

  Filter chips: All / Notes / Injuries / Physio / Travel. Each entry shows a

- **PROFILE** — locked in 2026-05-17 (see "Profile tab anatomy" section below for

  separate design pass), `PREFERENCES` (units, theme, notifications), `ACCOUNT`
  (email/password, connected providers, sign out), `DATA` (export, delete account).
  Exercise preferences (freeform v1, structured v2) live inside the athlete profile
  sub-route, not as a separate Profile item.

### Workout cards — primary types

Every activity card = primary type + subtype. Cards show a high-level summary on Today;
tapping routes to `/workout/[id]` for the full detail + log interaction. Card design and
detail-page design are separate passes — design the cards first.

| Primary type | v1 subtypes | Notes |
|---|---|---|
| Running | easy/recovery, tempo, threshold, intervals (VO2), hills, long | Attribute: terrain (road / trail / mixed). HR zones are the primary metric, then duration, then terrain/subtype-specific (pace for easy/long; vert for trail; structured segments for intervals) |
| Strength | upper, lower, full body, push, pull (subtype list TBD) | Logging is flexible: set-by-set or whole-session, same UI |
| Physio | general for v1 — specific exercise lists per session | Captures exercises done, sets/reps, pain level (1–10) |
| Cross-training | cycling, swimming, hike, other | One v1 card; split into primary types in v3. The **Hike** subtype's card gets a richer treatment (vert prominent, time-on-feet, optional fueling) closer to a long-run card — training hikes with significant vert are real training, not just recovery. Promote to its own primary type in v2 if usage warrants. |
| Mobility / movement prep | pre-run activation, daily mobility, recovery flow (TBD) | Pending: whether "warm-up" = within-workout segment vs. standalone routine |
| Rest | — | Not a card; a state of the Today section |

### Card display pattern

- Primary metric: HR zone (`Z3–Z4`, `Z4 sustained`)
- Secondary: Duration (`~60 min`)
- Tertiary: subtype/terrain-specific (pace, vert, distance, structured intervals)
- Planned + actual displayed together for comparison once logged
- Em-dash mono sub-label: `— TRAIL · TEMPO`, `— ROAD · LONG`, `— STRENGTH · LOWER`
- All numerals in Geist Mono, tabular

### Detail / log page (`/workout/[id]`)

Page structure (top to bottom): back link → header label → title + one-line description →
key metrics row → STRUCTURE section → WHY section (Claude-generated, specific to this
user's plan) → glossary link → LOG section (inline) → NOTES → log actions (sticky bottom
on mobile).

- Tap a card → routes to this full detail page (not inline expansion — better for
  back-stack, deep-linking, and the substantial logging interaction)
- Logging is inline on this page, not a separate route or modal. Falls back to a
  separate log step only if the inline approach proves too chaotic in practice.
- Planned + actual displayed together for comparison once logged.

LOG section variants by primary type:

- **Running**: actual duration / distance / vert / avg HR (required) +
  optional time-in-zone breakdown (Z1–Z5 minute inputs)
- **Strength**: Option D — collapsible exercise rows. Collapsed row marks the whole
  exercise done at planned values; expanding the row shows each set with editable
  reps + weight + an "add set" affordance for going beyond planned. User picks
  granularity by interaction, not by a mode toggle. Each exercise row displays
  an **equipment label** (e.g., `BARBELL`, `DUMBBELLS`, `SMITH/DB`, `BODYWEIGHT`)
  so the user knows what they need. A `— WARM UP FIRST` callout in emerald
  appears above the working sets, listing warm-up exercises (e.g., goblet squat,
  glute bridge, working-set ramps) so they're not buried inside the main work.
- **Physio**: each prescribed exercise + pain-level slider (1–10) with a
  **color gradient** (green at low pain → amber/orange at higher pain) for
  at-a-glance pain monitoring + per-exercise note input. EQUIP data tile in
  the metrics row surfaces what equipment the prescription needs (e.g., "Band").
- **Cross-training**: duration + perceived effort + optional HR
- **Mobility**: minimal — done toggle + optional duration

**Custom items in the LOG section (v1):** every LOG variant supports user-added
items beyond what Claude planned, and removal of planned items the user skipped
or substituted. Itemized LOG variants (Strength / Physio / Mobility) get a
`+ Add exercise / movement` affordance plus a subtle `× remove` per row;
single-block LOG variants (Running / Cross-training) get a `+ Add segment / activity`
affordance for unplanned additions. User-added items are visually distinguished
from Claude-planned items via a `USER-ADDED` mono tag on the inline add form
and (potentially) on the saved row. Removed items stay visible with strikethrough
+ `UNDO` link so the user can audit what was planned vs. what they did.

**Exercise preferences (v1 → v2 evolution):**
- *v1*: one-time swaps via the add/remove affordances above. Persistent
  preferences ("I always prefer split squats to walking lunges") are written
  as Journal notes; Claude consumes them on the next regeneration.
- *v2 (planned)*: structured "Exercise preferences" surface in Profile where
  users declare swap rules explicitly. Plus "Saved routines" templates the
  user can apply to any strength slot.
- *v2 (also planned)*: when removing/swapping an exercise on a workout,
  prompt the user — "Just for today, or as an ongoing preference?" — and
  auto-write a Journal note for the ongoing case.

**External strength program (deferred to v2):** "Strength: external" toggle
in Profile/wizard that tells Claude to stop generating strength workouts.
Users following 5/3/1, StrongLifts, or a coach-designed program use Vert
for running coaching + adherence logging only. Pairs with the Saved Routines
feature for users who want to apply their own templates to scheduled slots.

Glossary scope: workout types and subtypes only (tempo, threshold, long run, easy run,
hills, intervals; upper/lower/full strength categories). Does NOT cover individual
exercises — squats, RDLs, etc. are commodity movements; form questions point users
elsewhere. Primary entry: inline link from the WHY section in each workout drill-down.
Secondary entry: sub-route in Profile for top-down browsing.

### Regeneration result page

The screen the user lands on after triggering a plan regeneration. The most important
screen for the "adaptive AI" promise — if it feels like a black box, the value prop
collapses.

Entry points: Today's REGEN affordance · Plan tab's "Regenerate plan" CTA ·
auto-triggered after submitting an injury report or significant Journal note.

Page structure (top to bottom):
1. Header — back link, settings
2. State title in mono uppercase ("— PLAN UPDATED · just now")
3. Summary card framed as `— FROM YOUR COACH` — 2–3 sentences in plain
   English from Claude (the "coach voice"). The explicit framing establishes
   the AI as a coach rather than a system output.
4. Change badges — high-level moves as pill-style tiles with semantic colors:
   emerald tint for SHIFTED and ADDED, amber tint for REDUCED (signals a
   negative change without alarming). E.g., `SHIFTED · Sat → Sun`,
   `REDUCED · weekly volume −8%`, `ADDED · 2× calf strength`.
5. "— WEEK BY WEEK" diff — single-column with highlights on mobile; side-by-side
   OLD vs NEW available on desktop. Unchanged workouts dimmed; changed workouts
   in emerald with before → after.
6. "— BASED ON" — exactly what Claude considered (recent logs, notes, injury
   reports, race target, phase). Expanded by default for the first ~3 regenerations
   a user sees, then auto-collapses after the pattern is familiar.
7. Action bar (sticky bottom on mobile) — three actions:
   - **Accept new plan** (primary, emerald CTA)
   - **Regenerate again** (outline) — opens the universal Regenerate sheet
     (see "Universal Regenerate sheet" section below)
   - **Keep current plan** (text link, dim) — friendlier wording than
     "Discard"; tells the user explicitly what the action does

Accept model: **explicit accept**. Regenerations don't auto-commit. User reviews,
then accepts. Critical for trust — "adaptive" doesn't mean "out of my control."

States: A. Generating (full-screen atmospheric loading, 5–15s) · B. Result view ·
C. Minor change (collapsed diff, reassuring summary) · D. Accepted brief
confirmation (auto-routes back to Today) · E. Error (Claude API failed, plan
unchanged, retry CTA).

Error-state pattern (extends to other error states across the app): use
athletic-vocabulary framing rather than "Error" or "Failed" copy. The
regen error uses `— REGENERATION · REST DAY` with the title "Looks like our
servers are having a rest day." Body: "Your plan is safe and unchanged. Give
us a minute — we'll be back at it shortly." Plus a debug info row at the
bottom in small mono (`ERROR · NETWORK · REQ #2A8F`) for support diagnostics.
Apply this voice to other error states: offline (`— SIGNAL LOST`), sync
failed, plan-generation failure during wizard, etc.

The unified surface where the user tells Claude things to consider — feelings,

1. Header — settings cog top-right

## Decisions made 2026-05-20 (plan generation quality pass)

Session goal: audit and tighten the Claude plan-generation pipeline. Three
intertwined problems addressed in one change: the system prompt was vibes-level
methodology, the wizard collected ~30 fields but only ~12 were reaching Claude,
and a non-deterministic model could occasionally produce structurally broken
plans (missing days, no race on race day) with no enforcement.

### System prompt rewrite (`lib/claude.ts` SYSTEM_PROMPT)

Replaced the original "progressive overload, hard/easy" prose with explicit,
quantitative coaching rules the model can self-check against:

- **Named periodization phases** — BASE (40-50% of window) / BUILD (30-40%) /
  PEAK (10-15%) / TAPER (final 2 weeks, non-negotiable). Includes compressed-
  window handling (<12 weeks, <6 weeks, <3 weeks taper-only).
- **Hard quantitative rules** — 80/20 easy/hard distribution, 10% weekly volume
  cap, long run ≤30-35% of weekly volume (≤25% of race distance for ultras),
  48-hour spacing between hard sessions, no back-to-back hard days.
- **Race-specificity translation** — elevation gain → vert hikes / downhill prep,
  terrain → trail-specific runs, heat / climate prep, intent (competitive vs.
  finish strong vs. just finish) → quality density.
- **Injury-handling tiers** — acute / chronic flare risk / severe (recent surgery,
  stress fracture). Each tier has explicit volume + cross-training prescriptions.
- **Adherence-reading thresholds** — ≥80% / 50-79% / <50% completion bands with
  defined responses. Recent skipped long runs flagged as highest-priority signal.
- **B/C race handling** — no quality sessions in the 3 days before, no hard
  session within 2 days after.
- **Strength prescription by phase** — 2× in BASE+BUILD, 1× in PEAK, none in
  taper week -1. Injury-specific overrides (Achilles → heavy slow calf raises;
  hip/knee/IT band → glute/hip stability).
- **Session duration accounting** (added later same session, based on user
  testing feedback) — explicit per-exercise time budgets so Claude prescribes
  wall-clock time, not work time. Compound lifts ~12 min each, accessory
  ~8 min, isolation ~5 min, plus 5-8 min warm-up + 3-5 min cool-down.
  Mobility pacing: ~60-90s per movement (15 min = 6-8 movements, 20 min =
  8-10, 30 min = 12-15). Running/cycling/swimming wall-clock = work time.
  Fixes a ~30-40% under-estimate observed when actually using cloud-generated
  plans.
- **One-shot coach voice example** for the summary field so output style is
  anchored, not improvised.

### Wizard → prompt data flow expansion (`lib/claude.ts` formatProfile)

The wizard captured roughly 30 fields; the prompt only sent about 12. Audited
and threaded the rest through:

- Self-rated fitness (1-5 with human labels: "just starting out" through
  "highly trained, competitive")
- Years running, years doing ultras, ultras completed, longest race ever
  (distance + name + date)
- Sleep hours, stress baseline (1-5 with labels), chronic conditions
- Age, sex, body weight (with unit handling)
- Outdoor terrain access, training days available, preferred strength frequency
- B/C races (entire `OTHER RACES` prompt section added)

**Removed:** dead `easy_pace` field handling. The wizard doesn't capture an
easy pace and the prompt was being fed an empty string. Saved as a future
follow-up: add a 5K-time proxy question to the wizard so Claude can prescribe
numerical paces, not just qualitative effort. Tracked in memory.

### Structural plan validator (`lib/plan-validation.ts` — new module)

Lightweight validator runs after Claude returns. Three hard checks + one soft
warning:

- **`missing_dates` (error)** — every date from start through race day must
  have ≥1 workout
- **`no_race_day_run` (error)** — race day must include at least one workout
  of kind `run`
- **`dates_before_start` (error)** — no workouts dated before the start date
- **`long_run_in_taper` (warning)** — no long run in the final 7 days; logged
  only, never throws

Deliberately conservative scope. Quantitative rules (80/20, 10% volume cap,
etc.) stay prompt-only for v1 — adding them to the validator now risks rejecting
acceptable plans before we've seen the real failure distribution. Graduate
softer rules from prompt-only to validated as production data accumulates.

### Auto-retry-once on validation failure (`generateTrainingPlan`)

When validation finds error-severity issues, the system:

1. Logs the issues to the server console
2. Replays Claude's previous tool call back to itself as conversation context
3. Sends a `tool_result` with `is_error: true` listing what was wrong
4. Asks Claude to re-submit via `submit_training_plan`
5. If the retry also fails, throws (the action layer catches → error screen
   with Retry button)

Replaying the prior attempt as conversation context is the key detail — Claude
sees its own previous answer, so the retry is a targeted fix, not a fresh
generation. Costs one extra API call (~$0.05) on failure, absorbs most
transient model errors silently.

- Full suite: 97/97 tests passing. Files touched typecheck clean (pre-existing

## Decisions made 2026-05-20 (adherence summary + history windowing + prior summary)

Second pass on plan-generation quality, focused on what the regen prompt sees
when there's accumulated history. Goal: stop sending the entire training cycle
as a wall of raw workout lines and start sending high-signal pre-computed
patterns alongside recent detail.

### Adherence summary block (`computeAdherence` + `formatAdherenceSummary` in `lib/claude.ts`)

Pre-computes the patterns Claude was previously expected to extract from raw
data. Renders at the top of the `WORKOUT HISTORY` section as:

- **Rolling completion rates**: last 7d / 14d / 28d / cycle-to-date, each as
  "X/Y completed (Z%), N skipped, M unlogged"
- **By-kind breakdown** for the last 28 days: runs / strength / mobility
  completion ratios so Claude can spot "all 3 strength sessions skipped" type
  patterns at a glance
- **Most recent skipped workout** with date + kind + title (recent skipped long
  runs are flagged as highest-priority signal in the system prompt)
- **Skip clusters**: contiguous date ranges of ≥2 days with zero completed
  workouts, often signalling travel or injury onset. Multi-workout days with
  mixed status (one skipped, one completed) do NOT count as skip days

### History windowing (refactored `formatHistory`)

Detailed per-workout lines now limited to the last 28 days. Anything older
gets compressed into a single rolled-up block:

```
EARLIER IN THIS CYCLE (rolled up — not shown per-workout)
- Range: 2026-02-15 → 2026-04-19
- 47/61 completed (77%), 10 skipped, 4 unlogged
```

Reasoning: 28 days covers ~4 weeks (one training block including a cutback),
which is what Claude actually needs at fine resolution. Older context is
better expressed as adherence rates than as raw lines.

`formatHistory` signature changed to take `today: string` so windowing math is
deterministic. `buildUserPrompt` now threads `args.startDate` through.

### Journal `consumed` flag verified (no fix needed)

`commitPlan` in `app/actions.ts` correctly flips `journal_entries.consumed`
from `false → true` after a successful preview commit. Order: RPC commit →
mark preview accepted → flip journal. Comment in code explains the safety
ordering: if the flip fails, the next regen still sees "NEW" entries (safer
than losing them).

**Known small edge case** (not fixed this round): if a user creates a journal
entry between preview generation and acceptance, that new entry gets marked
consumed without Claude ever having seen it. Fix would be to record which
entry IDs were passed into each preview and only flip those on commit. Low
priority because the workflow is rare and the next regen would still consume
the entry.

### Previous coach summary as continuity context (`formatPreviousSummary`)

`plan_previews` table retains `generation_summary` on accepted rows (status
flips to `'accepted'` but the row stays). Added a new `getLatestAcceptedSummary()`
query in `lib/supabase/server.ts` and threaded it through `previewPlan` →
`generateTrainingPlan` → `buildUserPrompt`.

Renders just before `PLAN PARAMETERS` as:

```
PREVIOUS COACH MESSAGE (what you told the athlete in the last accepted regen —
use for continuity, but feel free to revise direction if recent context warrants):
"You missed two of last week's runs — pulled back this week's volume..."
- SHIFTED Sat long → Sun
- REDUCED weekly volume −8%
```

Framed explicitly as context, not directive, so Claude can build on the prior
direction or deliberately diverge. Skipped for the initial wizard-generated
plan (no prior summary exists) and for users whose previews were all discarded
rather than accepted.

### What was discussed and parked

**Prescribed-vs-actual deltas** (e.g., "ran 8% faster than prescribed"):
high-signal but requires structured prescription fields on the tool schema
+ workouts table to do cleanly (parsing free-text `details` is fragile). The
clean fix is additive — `prescribed_duration_min`, `prescribed_distance_km`,
`prescribed_elevation_gain_m`, `prescribed_target_zone` as new optional fields
on each workout. Saved as memory follow-up `project_structured_prescriptions.md`
with explicit evaluation step: before building, capture 3-5 real regen prompts
as fixtures, generate plans both with and without delta signals, and only ship
if there's a clear plan-quality improvement. Don't build it just because it's
clever.

### Test coverage

- `lib/claude.test.ts`: 34 tests passing (up from 19). New describes for
  `computeAdherence` (windowed rates, by-kind, most-recent-skipped, skip
  clusters), `formatAdherenceSummary`, `formatPreviousSummary`. Updated

## Buy vs. Build philosophy (decided 2026-05-17)

**Principle:** Buy the boring stuff, build the differentiated stuff. Anything that isn't part of
Vert's secret sauce should be a third-party service. Off-the-shelf services are usually more secure,
more reliable, and have edge cases solved that we'd otherwise hit at 2am.

**Note on WorkOS:** considered and rejected. WorkOS is for B2B SaaS that needs enterprise SSO
(SAML, SCIM, Okta/Azure AD integration). Vert is consumer software — Supabase Auth covers what we
actually need (email/password, magic links, OAuth providers).

## Race + athlete field spec (locked in 2026-05-18)

What Claude consumes to generate a plan. Wizard captures only must-haves
to keep first-time setup fast; Profile sub-routes capture must-haves +
nice-to-haves for richer plans over time.

### Race fields (per race, with ABC priority)

Race info is a race *calendar*, not a single race. Sub-line "Sorted by
priority — A races first." explains the ordering. Each race card shows
a priority badge with the letter in a colored square: A is emerald-filled
("PRIMARY TARGET"), B is emerald-outlined ("TUNE-UP RACE"), C is dimmed
("TRAINING-GRADE"). Past/completed races show a `COMPLETED` badge with
strikethrough on the race name. ABC prioritization following standard
ultra periodization:
- **A race** — the primary target. Plan builds toward this. Usually one
  per cycle.
- **B race** — secondary, raced hard but not peaked for. Used as a
  tune-up or fitness test.
- **C race** — training-grade. Used as a workout / training stimulus
  without a taper.

**v1 must-have (per race):**
- Race name
- Race date
- Distance (km/mi per user units pref)
- Elevation gain (m/ft per user units pref)
- Terrain (chip multi-select: trail, road, mountain, technical, mixed)
- Priority (A / B / C)
- Target finish time (optional; courses vary too much for precision)
- Effort intent (Competitive / Finish strong / Just finish / Training-grade)

**v1 nice-to-have (per race):**
- Elevation loss (often very different from gain on point-to-point;
  matters for quad damage planning)
- Cutoff time (intermediate cutoffs not just total)
- Climate / temperature expectation (hot / cold / altitude)
- Course profile type (point-to-point / loop / out-and-back / multi-stage)
- Self-supported vs. aid-station-supported (affects fueling strategy)

**v2+:**
- Course elevation profile (sparkline from GPX import or manual entry)
- Aid station spacing
- Race-specific strategy notes

### Athlete profile fields

**v1 must-have:**
- Current weekly running volume (per user units pref)
- Current weekly running time (hours)
- Longest run in the last month (informs fitness floor)
- Years of running
- Years of ultra-specific training
- Number of ultras completed (chip: 0 / 1–3 / 4–10 / 10+)
- Longest race ever completed — **distance** is the primary input (number
  input with unit suffix matching the user's global units preference, NOT
  a per-field unit toggle); optional race name (text) and optional
  completion date. Distance is what Claude actually uses; the name is for
  the user's reference.
- Self-rated fitness (1–5 scale, each with a description)
- Age
- Biological sex — **optional field**, options: Male / Female / Other /
  Prefer not to say. Relevant for plan generation research (recovery
  rates, VO2max, fueling, etc.) but always optional.
- Body weight (useful for fueling/hydration recommendations —
  grams of carbs per hour scales with weight)

**Intentionally excluded from v1:** height. It has no meaningful impact
on training plan generation (weight matters for fueling; height does not).
Keeping the field count lean.

**v1 nice-to-have:**
- Max HR (if tested or watch-derived)
- Resting HR (fitness indicator)
- Lactate threshold HR (if tested)
- VO2max estimate (from a watch or test)
- Recent race results (last 2–3: date, distance, time)

**v2+:**
- LT pace (from formal test)
- Vertical-specific weekly volume (some weeks are flat, some big-vert)
- HRV trends (heart rate variability — recovery indicator from connected wearables)
- Menstrual cycle phase tracking (relevant for performance and recovery research)
- Surgical history (structured input for major medical events that affect training)
- Detailed medication tracking (for users on training-impacting medications)

### Health & body fields (v1)

- Past injuries (freeform or structured chip list of body parts + notes)
- Current injuries (read-only reference to Journal injury reports —
  managed in Journal, surfaced here for context)
- Chronic conditions (asthma, diabetes, heart conditions, etc. — for
  safety even if Claude doesn't optimize for them)
- Sleep average (5 / 6 / 7 / 8 / 9+ hours per night)
- Stress baseline (1–5 scale)

### Lifestyle & training context

**v1 must-have:**
- Weekly hours available for training
- Typical training days (chip multi-select: M T W Th F Sa Su)
- Long run day preference (usually Sat or Sun)
- Quality day preference (Tue / Wed / Thu) — with helper text:
  "Your hardest workout of the week — tempo, threshold, intervals, or hills."
- Strength session frequency (none / 1× / 2× / 3× per week)

**v1 nice-to-have:**
- Time of day preference (morning / lunch / evening / no preference) — with
  helper text: "When do you usually fit in your workouts?"
- Job type (sedentary office / standing / physical / shift work) —
  affects recovery capacity

**v2+:**
- Family/caregiving load
- Travel frequency
- Climate where the user lives

Time zone: auto-detect from browser/system in v1. No explicit field.
v2 may add a manual override in Profile preferences if users need it.

### Equipment & environment (v1)

- Gym access (yes / no / limited)
- Equipment available (chips: treadmill, indoor trainer, weights, pool, none)
- Outdoor terrain access (chips: trails nearby, hills nearby, mountains nearby, flat only)

### Training preferences (v1)

- Cross-training preferences (chips: cycling, swimming, hiking, rowing, none)
- Freeform training preferences textarea (v1 home for "prefer X to Y";
  becomes structured swap rules in v2)

### Units handling (single Metric/Imperial toggle, global, not per-field)

One global units preference lives in Profile → Preferences as a single
toggle: **[Metric] [Imperial]**.

- Metric: distances in km, elevation in m, body weight in kg
- Imperial: distances in mi, elevation in ft, body weight in lb

The user picks one measurement system and everything in the app follows.
No per-system splits (e.g., miles + kg), no per-field toggles. The
edge case of "miles for running but kg for weight" is rare enough to
defer to v2+ if anyone requests it.

Every distance, elevation, and weight field throughout the app respects
this preference and shows the unit as a suffix only. Internal storage
stays metric (km, m, kg); display conversion happens at render time
based on the preference.

### Wizard vs. Profile split

The wizard captures **must-have fields only** to keep first-time setup
fast. The wizard ends with a note pointing to Profile: "Add more detail
about your training context in your Profile later for a more personalized
plan."

Profile sub-routes (`/profile/athlete`, `/profile/race`) capture
must-haves + nice-to-haves. Users who want a richer plan can fill in
nice-to-haves at their own pace.

### Wizard polish patterns (locked in via Claude Design review 2026-05-19)

Three UX patterns from the wizard are worth carrying forward to other
forms across the app:

- **`WE'LL USE [METRIC ⌄]` indicator** on the Welcome screen below the
  CTA. Surfaces the auto-detected measurement system with a tappable
  dropdown to switch. Same affordance pattern can apply anywhere a
  global preference defaults silently and the user might want a quick
  override.
- **Contextual description per selected option** — e.g., self-rated
  fitness chip 3 reveals "Consistent training" below the chip; selected
  effort intent reveals a one-line description ("Finish strong — Race
  for a strong personal effort."). Adds learning context as the user
  makes choices. Reduces ambiguity without cluttering the form by
  default.
- **`MULTI-SELECT *` indicator** marks required chip multi-select fields
  with the asterisk so the required-field convention is consistent
  across single-input and multi-input fields.

### Plan generation quality improvements (backlog, prioritised)

The Claude-generated training plans can be improved incrementally. Work through these in order:

1. ✅ **System prompt improvements** (done 2026-05-20) — periodization phases now
   explicit (BASE / BUILD / PEAK / TAPER with % allocations and compressed-window
   handling), quantitative rules (80/20, 10% volume cap, long-run % caps, 48-hour
   spacing), race-specificity translation rules, injury-handling tiers,
   adherence-reading thresholds. See "Decisions made 2026-05-20" subsection above.

2. 🟡 **Few-shot examples in the prompt** (partial — done 2026-05-20) — one coach-voice
   example added to the system prompt for the summary field. Worth extending later
   with 1–2 full example training weeks if output consistency still drifts.

3. **RAG database** (medium effort, v3+) — build a knowledge base of training methodology
   documents (Lydiard base building, 80/20 research, race-specific prep guides) and retrieve
   relevant content at plan-generation time. Worth exploring once real user feedback indicates
   plans aren't specific enough. **Note (2026-05-20):** discussed in this session and parked.
   Copyright on published plans (Daniels, Hansons, Pfitzinger, Koop, Roche) is the key
   blocker. Better path is a self-authored structured methodology library + template
   scaffolds + a data flywheel from validated user plans — those become the basis for
   either RAG or fine-tuning later, with no IP exposure.

4. **Model fine-tuning** (high effort, skip for now) — requires a large labeled dataset of
   good/bad plans, is expensive to run, and gains over strong prompting are usually marginal
   for a well-scoped task. Revisit only if the above options are exhausted.

5. ✅ **Structural plan validator + auto-retry-once** (added 2026-05-20) — validator
   in `lib/plan-validation.ts` catches missing dates, missing race-day run, and
   past-dated workouts. `generateTrainingPlan` retries once on failure by replaying
   the prior attempt as conversation context. Costs ~$0.05 on retry, absorbs most
   transient model errors silently. See "Decisions made 2026-05-20" subsection above.

6. **Model selection eval for plan generation + regeneration** (open, added 2026-05-20)
   — evaluate which Claude model strikes the right balance of plan quality and cost
   for both initial generation and regeneration. Approach: test candidate models
   (Haiku 4.5, Sonnet 4.6, Opus 4.6) against each other on a fixed set of
   representative wizard inputs + regeneration scenarios, then have Opus 4.6 judge
   the outputs against a **rubric** (not plan-to-plan comparison with published
   plans). Goal: a defensible default model per call site (initial generation may
   warrant Opus; regeneration may be fine on Sonnet or Haiku) that keeps plan
   quality high while keeping per-user API cost sustainable.

   **Why rubric, not plan-to-plan comparison:** benchmarking model outputs against
   specific published plans (Daniels, Hansons, Pfitzinger, Koop, Roche) would
   optimize the model toward replicating copyrighted structure — same IP exposure
   problem that parked the RAG approach. Methodology and training principles are
   not copyrightable; specific expressions are. So we derive a rubric from
   public-domain methodology and free/freely-shared sources, then score against
   the rubric.

   **Rubric dimensions (initial draft, to refine):**
   - BASE-phase weekly volume progression (≤10% week-over-week, cutback every 3–4 weeks)
   - Long-run as % of weekly volume (target 25–35%, capped)
   - 80/20 easy/hard polarization across the week
   - Taper structure appropriate to race distance (3-week marathon taper, etc.)
   - Race-specificity session placement (MP / threshold work in final 8 weeks)
   - Spacing between hard sessions (≥48 hours)
   - Injury-handling tier appropriateness when injury flag is set
   - Periodization phase coverage (BASE / BUILD / PEAK / TAPER allocations)
   - Coach voice quality on the summary field

   **Sources for deriving the rubric:** Hal Higdon's free plans, David Roche's Trail
   Runner Magazine columns, Jason Koop's koopcast blog, Steve Magness's writing,
   academic literature on polarized training (Seiler, Stöggl). Books (Koop, Daniels,
   Pfitzinger) are fine as inputs to Ben's own thinking when authoring the rubric —
   they should not be quoted into the rubric or eval prompts.

   **Output:** a Markdown eval report with per-model scores per dimension, total
   cost per generation, and a recommendation per call site.
